# Doc77 多语言化（i18n）设计方案

- **日期**：2026-07-16
- **状态**：已评审通过（brainstorming 阶段）
- **范围**：全部六个用户可见层 —— Web UI、CLI、API 错误消息、MCP/AI tool descriptions、AI system prompt、Electron 托盘

## 1. 背景与目标

Doc77 当前为纯中文软件，用户可见中文字符串约 400-600 条，分布在 5 层约 28 个源文件中（Web UI 占约 70%）。本次改造目标：

1. **中英双语适配** —— zh-CN 与 en-US 内置，随应用分发，零网络依赖
2. **可扩展任意语言** —— 通过外部语言包文件（`~/.doc77/locales/<lang>.json`）支持任意语言，本次实现目录扫描与加载逻辑，下载命令（`doc77 lang install`）留待未来
3. **算无遗漏** —— 建立系统化的词条提取流程与覆盖率校验脚本，并纳入 CI 防止回归

### 非目标

- 不实现语言包下载命令 / 下载 UI（仅预留目录约定与加载逻辑）
- 不引入复数规则、日期本地化等高级 i18n 功能（zh/en 场景不需要）
- 不改动文档内容翻译功能（`packages/core/src/translate/`，那是翻译用户文档的 Opus-MT 引擎，与 UI i18n 无关）
- 代码注释、`docs/` 项目文档不翻译

## 2. 技术选型

**自研轻量 i18n 模块**（约 100 行），不引入 i18next 等框架。

理由：

- 项目前端为纯原生 JS、无 bundler，后端奉行轻依赖、离线优先哲学
- 词条量中等（400-600 条），扁平 key-value + 占位符插值 + 回退链已满足全部需求
- 自研可让「外部语言包目录扫描」实现得最干净

已排除方案：i18next（需 vendor 浏览器版，为中等词条量引入重依赖）；构建期生成多套静态页面（无法支持下载语言包扩展任意语言）。

## 3. 核心模块设计

### 3.1 目录结构

```
packages/core/src/i18n/
├── index.ts          # t(), setLocale(), getLocale(), initI18n(), listLocales()
├── locales/
│   ├── zh-CN.json    # 内置中文词条（从现有文案提取）
│   └── en-US.json    # 内置英文词条（基准语言，回退目标）
```

所有下游包（ai / mcp / cli / electron）均已依赖 `@doc77/core`，i18n 模块由 core 导出即可全员共享，不新建 package。

### 3.2 API

```ts
initI18n(lang?: string): void
// 启动时调用：加载内置 JSON + 扫描 ~/.doc77/locales/*.json 合并

t(key: string, params?: Record<string, string | number>): string
// 查词条 + {name} 占位符插值
// 回退链：当前语言 → en-US → 返回 key 本身

setLocale(lang: string): void   // 运行时切换语言（写回 config 由调用方负责）
getLocale(): string             // 当前语言码
listLocales(): { code: string; name: string }[]
// 可用语言列表（内置 + 外部扫描），供设置面板下拉
```

### 3.3 词条格式

扁平 key-value JSON，key 命名规范为 `层.模块.语义`：

```json
{
  "_meta": { "code": "zh-CN", "name": "简体中文" },
  "cli.password.tooShort": "❌ 密码至少6位",
  "web.dashboard.greeting.morning": "早上好",
  "api.file.tooLarge": "文件超过 {maxSizeMB}MB 上限",
  "mcp.tool.writeFile.desc": "创建或覆盖文件内容（需审批）"
}
```

- 占位符使用 `{name}` 语法
- `_meta` 提供语言展示名，供 `listLocales()` 使用
- 外部语言包与内置格式完全一致；第三方语言包作者以 `en-US.json` 为模板

### 3.4 外部语言包合并规则

- 启动时扫描 `~/.doc77/locales/*.json`
- 外部文件语言码与内置相同时（如用户自放 `zh-CN.json`），**外部覆盖内置同名 key** —— 用户可自定义措辞
- 语言码为新语言时，注册为新可选语言

## 4. 语言选择与存储

策略：**全局单一设置 + Web 前端可覆盖**。

| 层 | 语言来源 |
|---|---|
| 全局设置 | config 表新增 `locale.language`（空值 = 自动检测） |
| 后端自动检测 | `locale.language` 为空 → 读系统 `LANG` / `LC_ALL` → zh 系 → `zh-CN`，其余 → `en-US` |
| Web 前端覆盖 | `localStorage.doc77_lang` 存在则优先于全局设置；设置面板提供「跟随全局 / 简体中文 / English / 外部语言…」下拉 |
| 前端首次访问 | 无 localStorage 且全局为自动检测 → 按 `navigator.language` 判断 |
| CLI | 读 config 的 `locale.language`（直接读 SQLite，server 未启动也可用），为空则看系统 `LANG` |

生效时机：

- 前端切换：即时生效（重新拉词条 + 重渲染文本）
- 全局切换：影响 API 报错 / AI / MCP；**MCP tool descriptions 在注册时固定，需重启服务生效**，设置面板注明此点

回退链：当前语言 → `en-US` → key 本身。`en-US` 为基准语言（符合开源惯例，第三方语言包以其为模板）。

## 5. 各层接入方式

### 5.1 Web UI（约 70% 词条）

- **HTML 静态文本**：元素加 `data-i18n="web.xxx"` 属性，页面加载时 `applyI18n()` 批量替换 `textContent`；`title` / `placeholder` 属性用 `data-i18n-title` / `data-i18n-placeholder`
- **JS 动态字符串**：`common.js` 提供全局 `t(key, params)`；页面启动时 `GET /api/i18n/:lang` 拉取合并后的词条表（带 ETag 缓存）
- **防闪烁**：词条表加载完成前 body 设 `visibility: hidden`（或内联首屏关键词条），避免英文用户先看到中文闪现
- 覆盖全部 5 个 HTML（含 mobile 两套）+ 12 个 JS 文件

### 5.2 CLI

- `packages/cli/src/bin/doc77.ts` 中所有 `console.log/error` 中文替换为 `t('cli.xxx')`
- CLI 启动时直接读 SQLite config 初始化 i18n

### 5.3 API 错误消息

- `app.ts` / `auth.ts` / `crypto.ts` / `ai-tools.ts` 报错替换为 `t('api.xxx')`
- 响应体同时携带稳定的 `code` 字段：`{ error: t('api.editNotAllowed'), code: 'EDIT_NOT_ALLOWED' }`
- 前端与测试断言 `code` 而非文案，一次性解决测试对文案的脆弱依赖

### 5.4 MCP / AI tool descriptions

- `packages/mcp/src/server.ts`（28 处）与 `packages/ai/src/tools.ts`（19 处）的 description 及 schema `.describe()` 改为 `t('mcp.tool.xxx')` / `t('ai.tool.xxx')`
- 注册发生在启动时，按当时全局语言固定
- `ai-tools.ts` 返回给 AI 的运行时文本（如「已加入审批队列」）同样走 `t()`

### 5.5 AI system prompt

- `SYSTEM_PROMPT` 整体作为一条长词条（`ai.systemPrompt`），zh / en 各写一版完整 prompt，**不逐句拼接**，保持 prompt 整体质量
- 原「中文优先」原则改为按当前语言注入「始终使用{语言}回复」
- `createClassifyPrompt()` 同样词条化

### 5.6 Electron 桌面端

- `packages/electron/src/tray.ts` 托盘菜单（「打开 Doc77」「退出 Doc77」）与 `main.ts` 中用户可见文本走 `t('electron.xxx')`（electron 包已依赖 `@doc77/core`）
- 语言来源与 CLI 相同：读 config 的 `locale.language`，为空则按系统语言检测

## 6. 防遗漏机制

1. **按文件清单逐个处理** —— 摸底已定位 28 个含中文源文件，实施计划中每个文件一个 task，做完打勾
2. **覆盖率扫描脚本** `scripts/check-i18n.sh`：
   - `grep -rP '[一-龥]'` 扫描 `packages/*/src`
   - 白名单排除：注释行、`translate/models.ts`（文档翻译模型显示名）、locale JSON 本身、测试文件
   - 扫描命中非白名单内容即退出非零
3. **key 完整性校验**：同一脚本比对 `zh-CN.json` 与 `en-US.json` 的 key 集合，双向 diff 为空才通过（防漏翻）
4. **纳入 CI 预检**：以 `pnpm check:i18n` 形式加入提交前检查链（`format:check && lint && check:i18n && build && test`），防止未来新增硬编码中文

## 7. 测试策略

- 现有约 6-8 处断言中文文案的测试（auth / ai-tools / mcp 集成 / ai tools）：改为断言 `code` 字段，或用 `t()` 取期望值
- 新增 i18n 模块单测：回退链、占位符插值、外部包合并覆盖、`_meta` 解析、`listLocales()`
- `live-e2e.test.ts`：中文 prompt 属产品行为测试输入，保持不变，基本不受影响

## 8. 词条翻译

- zh → en 翻译由 AI 完成（400-600 条），UI 短语参照业界惯例措辞（Save / Cancel / Approve / Settings 等）
- 完成后由用户抽查 `en-US.json`
