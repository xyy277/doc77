# doc77 能力扩展 Roadmap（增长视角）

> **状态:** 方向蓝图(Roadmap),非实施 spec — 记录探索结论与建造次序。每个方向后续单独走 spec → plan → 实施。
> **日期:** 2026-07-14
> **衡量尺子:** 增长 / 推广(新增用户、口碑传播、降低上手门槛)

## Context

本文档是一次"探索新方向"头脑风暴的沉淀。目标不是设计单个功能,而是**在增长视角下,把 doc77 今后的能力扩展方向理清、排序、记录**,为后续逐个立项提供依据。

doc77 现有能力基线:**本地优先、默认安全的文档预览器 + MCP 文件操作桥梁 + 可选 AI Agent**。渲染已相当完善(Markdown/Mermaid/PDF/code/图片/DOCX/XLSX、KaTeX、PlantUML、多 tab、面包屑),已支持 0.0.0.0 局域网访问与移动端 web 入口,已有 vendor 离线缓存机制。

**核心张力(贯穿所有增长决策):** doc77 主打"本地安全",而最能传播的功能往往要求"内容离开本机"。因此每个增长方向都要标注隐私张力,能守住定位的优先。

### 新方向(本次探索)
| 方向 | 增长机制 | 隐私张力 | 现有基础 |
|---|---|---|---|
| **A. 一键分享 / 发布** | **病毒传播** — 每个分享出去的文档带 doc77 品牌,别人看到即潜在用户 | **最大** — 分享=内容离开本机。分层降张力:①局域网临时链接 ②自包含 HTML ③公网发布(最重、最违背定位) | 渲染已足够漂亮;已支持 0.0.0.0 局域网 |
| **B. 零成本导入** | **消灭切换成本** — 一键导入 Obsidian vault / Notion 导出 / Markdown 文件夹,解析 wikilink 双链,吸存量用户 | 无 | "注册项目目录"模型天然契合 |
| **C. 精美导出** | **导出物即营销素材** — 一键导出精排 PDF / 长图 PNG / 自包含 HTML(带主题、mermaid、katex),截图级美感 | 无(导出是本地行为) | 已有完善渲染管线;与"轻量编辑"互补 |
| **D. 随身文档 / 移动端伴侣** | **跨设备 wow,易口口相传** — 扫码即在手机上读电脑文档 | 小(仅局域网) | 移动端 web 入口 + 0.0.0.0 绑定已具备 |

## 关键洞察:A + C 共用内核

**A 的"分享"第一期依赖 C 的"自包含 HTML"引擎** —— 两者共用同一内核(把一篇文档变成一个自包含、可携带的产物)。因此:

- **A + C 合并为一份内聚 spec:「导出与分享」** = 导出 PDF/长图/自包含 HTML + 局域网分享链接。第一期只做"自包含 HTML / 局域网链接",病毒性有了、内容还没真正上公网,守住定位。
- **B(导入)与 D(移动端)彼此独立**,各自单独成 spec。

## 技术可行性与性能

### 技术可行性评级
| 方向 | 可行性 | 关键点 |
|---|---|---|
| C 精美导出 | ⭐⭐⭐⭐⭐ | Electron 自带 Chromium:PDF 用 `webContents.printToPDF()`、长图用 `capturePage()`,零新依赖;自包含 HTML = 内联 CSS + base64 图片,纯 Node。纯 CLI/web 模式无 Electron 时 PDF 需退回 headless(较重) |
| A 局域网分享 / 自包含 HTML | ⭐⭐⭐⭐⭐ | 已绑 0.0.0.0 + 已有 auth,≈ 只读 token + 一个路由 + UX,无新技术 |
| B 零成本导入 | ⭐⭐⭐⭐ | Obsidian vault = 文件夹 + `[[wikilink]]` 解析,复用"注册目录";Notion 导出格式较乱,中等 |
| D 移动端伴侣 | ⭐⭐⭐⭐ | 复用移动 web + 局域网,≈ 局域网 URL 二维码 + UI 打磨;约束:同一局域网 + 防火墙放行 |
| A 公网发布 | ⭐⭐ | 唯一架构级难点,见「公网发布的三档实现」 |

**小结:** C / A(局域网) / B / D 都是"UX + 少量解析"级别,无技术拦路虎;桌面版靠内置 Chromium 在导出/PDF 上天然占便宜。唯一真难点是 A 的公网发布。

### 性能与"变重"护栏
架构已把成本**按功能隔离**——不是"加一个功能全体变重"。守住三条现有纪律即可:
- **懒加载**:重依赖只在用到时加载(CodeMirror 仅编辑模式、mermaid/pyodide 走 vendor 按需、翻译引擎/导出渲染同理),预览页常驻只有核心 JS。
- **可选模块 + 按需下载**:ai/mcp 运行时按需装,翻译模型按需下,不进基础包。
- **Lite / Full 双版本**:重量级能力 opt-in 下载或仅进 Full 版。

运行期大头只有两处、且瞬时可释放:翻译推理内存(~200-400MB,翻完可卸载)、导出渲染 CPU。均"用时才付费"。

**护栏建议:** 给"常驻预览路径"设**性能预算**(预览页 JS 冷启动体积上限 + 首屏时间),每加一个能力强制走"懒加载 + capability flag",CI 盯住预算不越界 → 加再多功能,空载 doc77 仍是轻量预览器。

| 顺序 | 方向 | 理由 |
|---|---|---|
| 1️⃣ | **导出与分享(C + A)** | 天花板最高的增长引擎;C 打底、A 骑在 C 上;单独也有价值;隐私张力可用分层设计守住 |

已有的 editing / translation 按各自设计节奏推进,归入同一张能力蓝图。

## 公网发布的三档实现(方向 A)

**核心矛盾:** doc77 是本地桌面应用,而"公网可访问"要求内容能被没有你机器的人访问到。三档,越往下增长越猛、越重、越违背"本地安全":

| 档位 | 实现 | 要 doc77 跑服务器吗 | 隐私 | 增长 |
|---|---|---|---|---|
| **① 静态导出 + 用户自托管**(推荐起步) | 生成自包含 HTML / 静态站,用户自己丢 GitHub Pages / Netlify / 任意静态托管 | **不用**,零基建 | 守定位(用户自己发) | 分享页带 doc77 品牌 |
| **② 临时隧道**(可选进阶) | ngrok / Cloudflare Tunnel 式,临时开公网 URL 隧道回本机,关 app 即失效 | 不用持久托管(依赖隧道服务) | 内容仍在本机、链接临时 | "临时给人看一眼"体验好 |
| **③ doc77 自建托管** | 运营云服务,app 上传内容,给持久品牌直链 | **要**(跑 + 付 + 存内容 + 审核) | 最违背定位 | 最强 |

**结论:** 第一期做 **①(静态导出/自包含 HTML)+ 局域网链接**,完全不碰服务器(即「A 第一期」);要公网直链优先 **② 临时隧道**;**③ 自建托管不是一个功能,而是"要不要把 doc77 变成一门云服务"的商业决策**,成本 / 隐私 / 审核全来,最砸"本地安全"招牌——单独评估,不混进这波能力扩展。

## 隐私红线注记

增长功能中,**唯有 A 的公网发布档位与"本地安全"定位正面冲突**。建造 A 时必须:
- 默认从最安全档位起步(① 静态导出 / 局域网链接),② 临时隧道与 ③ 自建托管作为显式 opt-in 的更高档位。
- 任何"内容离开本机"的动作都需用户明确授权,不得默认开启。
- ③ 自建托管在动工前必须作为独立的商业决策评估,不随能力扩展默认纳入。
